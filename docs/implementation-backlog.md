# Token 潮汐（Token Tide）实施任务拆分

> 原则：先验证数据真实性，再开发排行榜；先做模块化单体，再按负载拆服务。

## 0. 里程碑

| 里程碑 | 周期 | 可演示结果 |
|---|---:|---|
| M0 可行性 PoC | 2 周 | 工具×模型采集矩阵、飞书动态 URL 签名、样本解析报告 |
| M1 内部 MVP | 4 周 | 20–50 人可安装上传、看个人页和排行榜 |
| M2 企业增强 | 3 周 | Cursor 企业同步、部门榜、数据质量后台、战报 |
| M3 正式上线 | 2 周 | 安全审计、灰度、监控、运维与回滚齐备 |

## 1. Phase 0：可行性 PoC

### P0-1 数据源样本与合规边界（P0）

- 负责人：采集端 + 安全。
- 收集 macOS/Windows、不同版本的脱敏样本，不把原文件提交到代码库。
- 定义字段白名单与禁止字段扫描器。
- 产出：`source-capability-matrix.md`、样本生成器、合规检查表。
- 验收：每个工具给出 A/B/C/D 精度结论和明确的上线/不上线建议。

### P0-1A 适配器 SDK 与兼容矩阵（P0）

- 定义 `Detect → Preview → Collect → Normalize` 接口、适配器清单和签名机制。
- 建立工具、操作系统、版本、模型字段、Token 分项、精度和最近验证日期的兼容矩阵。
- 第一批除 Codex/Claude Code 外，验证 Gemini CLI、OpenCode、Aider、LiteLLM/OpenRouter 或公司模型网关中至少两个来源。
- 验收：新增一个样例工具适配器不修改上传、计价、聚合和网页模块。

### P0-2 Codex 解析器 PoC（P0）

- 探测 CLI/桌面端本地数据位置和版本，解析实际模型原始标识、时间与 Token 分项。
- 构造增量游标、去重 ID、损坏行容错。
- 验收：真实脱敏样本重复扫描结果一致，不读取正文。

### P0-3 Claude Code 解析器 PoC（P0）

- 同 P0-2；额外验证缓存 Token 与子任务/子代理是否重复累计。
- 验收：与工具自身用量输出或账单抽样对齐。

### P0-3A 模型别名与价格 PoC（P0）

- 从各工具样本收集原始模型标识，建立 `observed_alias → canonical_model_id` 注册表。
- 对 GLM、DeepSeek、Claude、GPT 等系列验证精确版本、供应商、调用渠道和 Token 类型价格。
- 对 `dpV4`、`ops4.8`、`gpt5.5` 等未核实名称只保留观测原文，禁止猜测映射。
- 验收：未知模型仍计 Token、费用显示未知；映射后可追溯到价格版本并重新计算。

### P0-4 Cursor 能力 PoC（P0）

- 验证公司套餐与 Admin/Analytics API 权限、按用户粒度、时间窗口、金额字段。
- 个人/普通团队本地数据仅做只读验证，不抓包。
- 验收：确定企业 API 同步方案和非企业版降级方案。

### P0-5 Trae 能力 PoC（P0）

- 查验官方导出/API和本地结构化 usage 字段。
- 验收：无可靠 Token 时明确只进入请求/活跃榜，不能以估算补齐主榜。

### P0-6 飞书 PoC（P0）

- 创建测试应用，跑通 OAuth、租户与用户身份。
- 参考 `aidenzou-az/weather-signature`，验证飞书签名粘贴 URL 后实际抓取 `<title>`、description、OG 元数据的规则、缓存时间和刷新行为。
- 验证不可枚举个人 URL、撤销/轮换、匿名模板和过期数据提示。
- 验收：用户只配置一次 URL，后端统计变化后签名预览能在缓存窗口后更新；不暴露内部 ID。

### P0-7 方案评审门（P0）

- 产品、研发、安全、HR/法务、IT 管理员共同评审。
- Go 条件：至少两个来源达到 B 级、飞书登录跑通、禁止字段扫描通过。

## 2. Phase 1：MVP 产品与平台

### P1-1 仓库与工程基线（P0）

- Monorepo：`apps/web`、`apps/api`、`apps/collector`、`packages/contracts`、`infra`。
- CI：lint、unit test、schema compatibility、secret scan、跨平台构建。
- 本地开发：Docker Compose 启动 PostgreSQL/Redis/API/Web。

### P1-2 身份、租户与权限（P0）

- 飞书 OAuth 回调、token 安全存储、租户映射、会话管理。
- RBAC：员工/负责人/管理员/审计员。
- 验收：跨租户访问测试全部拒绝；撤销授权后会话失效。

### P1-3 设备配对与凭证（P0）

- 5 分钟一次性配对码、设备公钥、访问/刷新令牌、撤销设备。
- macOS Keychain 和 Windows Credential Manager 存储。
- 验收：配对码不可重放，撤销后上传立即失败。

### P1-4 统一事件与上传 API（P0）

- JSON Schema/Protobuf 合同、批量上传、字段白名单、幂等、限流。
- 异常事件隔离队列和逐条错误响应。
- 验收：重复上传、乱序上传、部分失败、未来时间和超大批次测试通过。

### P1-5 跨平台采集器外壳（P0）

- 单文件 Go CLI：`scan --preview`、`sync`、`doctor`，默认一次性运行并退出。
- `/connect` 页面按 macOS、Windows、Linux 给出签名安装包和一行命令，并提供脚本源码、版本、SHA-256 与权限说明。
- 实现设备授权，不在安装命令中携带长期令牌；默认用户目录安装且不要求管理员权限。
- 首次上传成功后提示“建议开启定时采集”，提供每 6 小时、每日、仅手动和不再提醒；未经主动同意不创建计划任务。
- Wails 托盘与后台同步作为可选组件，默认关闭。
- 验收：macOS Intel/Apple Silicon、Windows x64、Linux x64/arm64 首次同步正常；卸载不留下后台进程。

### P1-5A 网页接入状态与上传回执（P0）

- SSE/WebSocket 展示等待安装、已配对、扫描中、已上传、已聚合五段状态。
- 首次回执展示工具、模型数、事件数、Token、时间范围、计价覆盖率和被拒绝字段。
- 针对未运行、未授权、无日志、无 usage、网络失败、未知模型提供独立修复引导。
- 验收：用户不用查看终端日志即可在网页确认数据是否收到和为什么未收到。

### P1-6 Codex 与 Claude Code 生产适配器（P0）

- 将 PoC 解析器产品化，加入版本探测、游标、夹具和遥测。
- 验收：解析失败只影响单一来源；远程配置可停用坏版本。

### P1-7 价格与汇率引擎（P0）

- 模型注册表、版本化别名、调用渠道价格、Token 类型计价、未知模型队列、汇率快照。
- 三种成本类型分离，支持追溯和模拟重算；账号类型支持个人订阅、API 按量、公司统一账号和不确定。
- 个人订阅允许填写计划名、月费、币种和结算日；无法判断时只给 API 等价总价并标 `≈`。
- 总价按每个模型费用求和，并返回已计价 Token 覆盖率。
- 验收：边界日期、价格切换、别名归一、缓存价格、币种精度测试通过。

### P1-8 聚合与排行榜（P0）

- 按用户、工具、规范模型的日聚合与任意周期汇总；实现总榜、名次并列规则、累计潮汐值、海洋动物升级和迟到数据重算。
- 每个榜单用户返回模型 Token 占比、模型成本明细、周期活跃工具标签。
- 缓存与快照；名次变化基于上一个等长周期。
- 验收：组织时区、周一开周、月末/年末、自定义区间测试通过。

### P1-9 Web 首页与个人页（P0）

- 飞书登录页、竞技场总排行榜、用户下方模型占比条、姓名后工具标签、个人趋势与模型明细。
- 响应式、键盘可用、减少动态效果、空状态与精度提示。
- 验收：桌面主流浏览器可用，P95 查询时间达到设计目标。

### P1-10 匿名、展示名与隐私（P0）

- 稳定匿名化名、公开范围、最小群组人数、个人数据导出/删除入口。
- 验收：公开 API 不返回匿名用户真实姓名、飞书 ID 或邮箱。

### P1-11 签名生成器（P1）

- 今日/本周/本月模板，包含总 Token、总价、海洋动物等级、主力模型和更新时间。
- 为每位用户生成 128-bit 以上随机公开 slug；支持复制、预览、轮换和撤销。
- 动态输出 `<title>`、description、OG 元数据与详情页；为飞书缓存设计短 TTL、stale fallback 和刷新观测。
- 验收：签名 URL 不含任何内部 ID，撤销立即失效，缓存刷新时间有明确提示。

### P1-11A 海洋动物徽章与成长系统（P1）

- 完成小海星、小海马、水母、小丑鱼、海龟、海豚、章鱼、鲨鱼、虎鲸、蓝鲸 10 枚统一风格徽章。
- 输出 SVG 和 32/64/128/256 px 透明 PNG，包含暗色安全色、单色降级版与 alt 文本。
- 实现累计潮汐值、公式版本、升级记录、下一级进度和只升不降规则。
- 排行榜、个人页、升级动画、飞书 `<title>`/OG 图片共享同一动物资产版本。
- 验收：32 px 仍可辨识；飞书不显示图片时退化为动物 Emoji + 等级文本。

### P1-12 管理后台最小集（P1）

- 价格表、未知模型、客户端版本、适配器兼容矩阵、远程熔断、审计日志。

## 3. Phase 2：企业增强

### P2-1 Cursor 企业同步器（P0）

- 管理员密钥托管、日期分片、分页、ETag、限流退避、用户映射。
- 与客户端事件建立来源优先级，避免双算。

### P2-2 Trae 适配器（P1）

- 仅在 PoC 达到 B 级时加入 Token 榜；否则发布“活跃度适配器”。

### P2-3 后台自动同步与更新（P1）

- 用户主动开启后创建用户级计划任务，支持每 6 小时、每日和仅手动；设置页可随时关闭并清理任务。
- 定时器、离线队列、签名校验的客户端更新包；不得通过常驻注入进程采集。
- 分阶段灰度和一键停更。

### P2-4 部门榜与战报（P1）

- 部门聚合、最小人数限制、日/周飞书机器人卡片。
- 战报强调趋势、缓存效率和采用率，不只强调消耗量。

### P2-5 数据质量中心（P1）

- 解析成功率、未知模型率、迟到率、重复率、精度分布、修订审计。

## 4. Phase 3：上线与运维

### P3-1 安全与隐私评审（P0）

- 威胁建模、依赖与二进制扫描、渗透测试、日志脱敏、权限复核。
- 抽样证明禁止字段未离开本机。

### P3-2 灰度计划（P0）

- 5 人研发内测 → 20 人跨平台 → 50 人部门灰度 → 全公司。
- 每阶段至少运行 5 个工作日，满足成功率和数据一致性门槛再扩容。

### P3-3 可观测与告警（P0）

- API SLO、队列积压、聚合延迟、适配器失败、成本异常、OAuth 失败。
- 值班手册、故障分级、数据回放和回滚流程。

### P3-4 发布与回滚（P0）

- Web/API 蓝绿或滚动发布；数据库向前兼容迁移。
- 客户端签名、渠道、自动更新灰度、最低版本和紧急熔断。

## 5. 建议团队配置

| 角色 | 投入 |
|---|---:|
| 产品/设计 | 1 人 |
| Web 前端 | 1 人 |
| 后端 | 1–2 人 |
| Go/桌面端 | 1–2 人 |
| QA | 1 人 |
| 安全/IT/飞书管理员 | 兼职评审 |

在 4 名研发配置下，M0 到 M2 约 9 周；若只有 2 名研发，建议先砍掉 Trae、自动签名、部门榜和后台自动更新，MVP 保持在 6–8 周。

## 6. 第一轮迭代建议（可直接开工）

1. 建立 Monorepo、事件 Schema 与适配器 SDK。
2. 同时开展 Codex/Claude Code 模型识别、更多 CLI/网关适配、Cursor/Trae、模型价格注册表、飞书动态 URL 签名 PoC。
3. 用固定假数据搭出 `/connect` 接入状态页、排行榜和个人页交互原型。
4. 第二周末召开 Go/No-Go 评审，冻结 MVP 数据源和成本口径。
5. 第三周开始设备配对、上传、计价、聚合和网页的纵向闭环。

第一轮的成功标准不是“页面做完”，而是拿一台 Mac 和一台 Windows 机器，从真实本地用量安全地产生一条可追溯事件，并在网页正确显示 Token、成本、排名与精度。
